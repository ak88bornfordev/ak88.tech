Как я ускорил интернет-магазин на OpenCart: разбор запросов MySQL

// план запроса EXPLAIN сюда

Небольшой интернет-магазин на OpenCart, каталог около 40 тысяч товаров. Страница категории с фильтрами грузилась 3–4 секунды — для покупателя с мобильного интернета это почти гарантированный уход со страницы. Ниже — что помогло на самом деле, без переезда на другой движок.

Шаг 1. Смотреть в лог медленных запросов, а не гадать

Первым делом включил slow_query_log в MySQL и оставил магазин под обычной нагрузкой на сутки. Список медленных запросов сразу показал, что дело не в одном «тяжёлом» месте, а в фильтрации товаров по нескольким атрибутам одновременно — цена, наличие, категория.

Шаг 2. Составной индекс под реальный фильтр

-- было: полный перебор таблицы товаров при каждом фильтре
CREATE INDEX idx_product_filter
  ON oc_product (status, quantity, price);

Индекс именно в этом порядке полей — сначала то, что фильтруется всегда (статус товара), затем наличие и цена — сократил время выборки для страницы категории с 1.8 секунды до 60 миллисекунд.

Шаг 3. Кэширование дерева категорий

Дерево категорий в OpenCart по умолчанию пересчитывается почти на каждой странице. Вынес его в кэш на файловой системе с инвалидацией только при изменении категорий из админки — это убрало десятки повторяющихся запросов на каждый визит.

Шаг 4. Изображения и ленивая загрузка

  • Сжал изображения товаров без потери качества — вес страницы категории упал почти вдвое
  • Добавил loading="lazy" для карточек товаров ниже первого экрана
  • Убрал неиспользуемые JS-модули темы, которые грузились на каждой странице «на всякий случай»

Результат

Страница категории стала открываться за 700–900 мс вместо 3–4 секунд, а конверсия из просмотра каталога в добавление в корзину выросла — хотя это уже, конечно, эффект не только от скорости. Самый большой урок для себя: прежде чем что-либо менять в коде движка, нужно посмотреть, что реально происходит на уровне базы данных — обычно там и находится настоящая причина.
